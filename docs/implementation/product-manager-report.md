# PromptVault 产品需求拆解报告

> 产品经理：AI Agent
> 日期：2026-04-01
> 参考文档：`docs/功能规划_v2.md`

---

## 一、产品愿景再定义

**核心价值主张**：让 Prompt 工程从玄学变成科学

**目标用户画像**：
- 中文开发者团队（3-5人）
- 有 AI 应用开发需求，需要反复调优 prompt
- 关注 prompt 质量稳定性

**V1 核心路径**：写好 prompt → 测好 prompt → 知道哪个更好

---

## 二、User Story 拆解

### 2.1 基础设施层

#### US-001：异步任务队列

```
As a 开发者
I want 将耗时操作（批量测试、A/B测试、评测集生成）放入后台队列执行
So that 不用等待，页面保持响应，且任务状态可追踪
```

#### US-002：API 限流与配额管理

```
As a 开发者
I want 系统自动控制 AI API 调用频率和配额
So that 不会因为误操作或配置错误导致 API 限流或账单超支
```

#### US-003：AI 响应缓存层

```
As a 开发者
I want 相同输入的 AI 调用结果被缓存
So that 降低 token 消耗、减少响应延迟、节省成本
```

---

### 2.2 Prompt 编写体验

#### US-004：变量实时预览

```
As a 开发者
I want 在编辑 prompt 时，右侧面板实时渲染 {{variable}} 的预览效果
So that 所见即所得，快速验证变量替换是否正确
```

#### US-005：草稿自动保存

```
As a 开发者
I want 每次修改后自动保存草稿
So that 不丢失工作进度，减少手动 Ctrl+S 的负担
```

---

### 2.3 Prompt 测试层

#### US-006：Prompt 批量测试

```
As a 开发者
I want 上传多个测试用例，一次性对单个 prompt 执行批量测试
So that 高效验证 prompt 在不同输入下的表现
```

#### US-007：多轮对话流程测试

```
As a 开发者
I want 模拟多轮对话，验证 prompt 在上下文累积场景下的表现
So that 发现上下文遗忘、角色漂移等问题
```

#### US-008：A/B 测试 + 统计显著性引擎

```
As a 开发者
I want 对比两个版本的 prompt，通过序贯检验自动判断哪个更好
So that 用数据说话，而非凭直觉猜测
```

#### US-009：自动评测集生成

```
As a 开发者
I want AI 自动生成评测集，并支持人工补充修正
So that 不必手动编写大量测试 case，降低测试准备成本
```

---

### 2.4 Prompt 质量层

#### US-010：Prompt 质量评分卡

```
As a 开发者
I want 从 Clarity、Completeness、Example、Role 四个维度获取 prompt 评分
So that 量化了解 prompt 质量短板，有针对性优化
```

#### US-011：Prompt 回归检测

```
As a 开发者
I want 保存时自动检测 prompt 质量是否下降
So that 在发布前发现质量退化，而非上线后才知道
```

---

### 2.5 可观测性层

#### US-012：结构化日志 + AI 调用链路追踪

```
As a 开发者
I want 查看每个 AI 调用的 token 消耗、延迟、模型版本
So that 排查问题时有据可查，优化时有数据支撑
```

#### US-013：成本分析中心

```
As a 开发者
I want 查看 token 消耗月度趋势和成本分布
So that 了解 AI 成本去向，合理规划预算
```

---

### 2.6 效率增强层

#### US-014：场景标签

```
As a 开发者
I want 用场景标签而非分类组织 prompt
So that 搜索效率更高，标签更贴合实际使用场景
```

#### US-015：Chain-of-Thought 引导器

```
As a 开发者
I want 获得 CoT 推理类 prompt 的最佳实践模板
So that 快速写出高质量的推理类 prompt
```

## 三、MVP 范围定义

### 3.1 Must-have（V1 交付，6-8 周）

| US | 名称 | 关联功能 | 优先级 |
|----|------|---------|--------|
| US-001 | 异步任务队列 | P0 | 必须，基础设施 |
| US-004 | 变量实时预览 | P0 | 必须，核心编辑体验 |
| US-006 | Prompt 批量测试 | P0 | 必须，核心测试路径 |
| US-008 | A/B 测试 + 统计显著性 | P0 | 必须，"知道哪个更好" |
| US-009 | 自动评测集生成 | P0 | 必须，降低测试准备成本 |
| US-002 | API 限流与配额管理 | P1→P0 | **升级**，防止失控是底线 |
| US-011 | Prompt 回归检测（轻量版） | P1→P0 | **升级**，保存即检测 |

**Must-have 核心原则**：没有这 7 条，V1 无法形成闭环（写→测→评）

### 3.2 Should-have（V1 后期或 V1.1）

| US | 名称 | 关联功能 | 优先级 | 理由 |
|----|------|---------|--------|------|
| US-010 | Prompt 质量评分卡 | P1 | Should | 量化质量有强烈需求，但不影响核心闭环 |
| US-007 | 多轮对话流程测试 | P1 | Should | 高级测试场景，不是所有人第一版就需要 |
| US-003 | AI 响应缓存层 | P1 | Should | 降本有效，但不是 V1 能否成功的关键 |
| US-012 | 结构化日志 + 追踪 | P2→P1 | **升级** | 可观测性是团队协作的基础，应提前纳入 |

### 3.3 Nice-to-have（V1 完成后再做）

| US | 名称 | 理由 |
|----|------|------|
| US-013 | 成本分析中心 | 依赖数据积累，V1 数据量不足 |
| US-014 | 场景标签 | 可用分类替代，锦上添花 |
| US-015 | CoT 引导器 | 知识型功能，需要内容运营 |

## 四、优先级调整说明

### 4.1 国内团队场景适配

基于国内团队使用特点，做以下调整：

| 原优先级 | 功能 | 调整后 | 理由 |
|---------|------|--------|------|
| P1 | API 限流与配额管理 | **P0** | 国内 API 限流问题更普遍（MiniMax/阿里云/百度），且批量测试极易触发限流 |
| P1 | Prompt 回归检测 | **P0** | 国内模型更新频繁（厂商迭代快），回归检测是质量保障底线 |
| P2 | 结构化日志/追踪 | **P1** | 团队协作场景下，出问题时的可追溯性是硬需求 |

### 4.2 去掉的 P2 功能

| 原 P2 | 去/留 | 理由 |
|-------|--------|------|
| 成本分析中心 | 暂缓 | V1 token 消耗量级小，分析价值有限 |
| 场景标签 | 暂缓 | 用分类和搜索替代，够用 |
| 草稿自动保存 | **保留** | 体验优化，不增加实现复杂度 |

## 五、验收标准（DoD）

### US-001：异步任务队列

- [ ] 后台 goroutine 池消费 task 表中的任务
- [ ] 任务状态覆盖：pending → running → done/failed
- [ ] SSE 实时推送任务进度（0-100）
- [ ] 任务类型支持：batch_test、ab_test、eval_gen、optimize
- [ ] 服务重启后，running 状态任务恢复执行

### US-002：API 限流与配额管理

- [ ] 每个 AI Provider 独立配置 QPS 上限
- [ ] 超出 QPS 时任务进入等待队列，而非直接失败
- [ ] 配额用尽时有明确告警
- [ ] 支持按 prompt/用户维度设置配额上限

### US-003：AI 响应缓存层

- [ ] 相同 prompt + 相同变量值 → 命中缓存，直接返回
- [ ] 缓存 key 包含 model 版本，防止跨版本混淆
- [ ] 缓存 TTL 可配置（默认 24h）
- [ ] 缓存命中率在日志中可见

### US-004：变量实时预览

- [ ] 正则匹配 `{{variable}}` 语法
- [ ] 支持 `{{variable|default}}` 默认值
- [ ] 预览面板在变量值变化时 300ms 内更新
- [ ] 无变量时不显示预览面板
- [ ] 右侧 40% 面板布局

### US-005：草稿自动保存

- [ ] 每 10 秒检测内容变化，有变化则保存
- [ ] 草稿不产生版本历史
- [ ] 关闭页面时提示未保存内容
- [ ] 页面重新打开时恢复草稿

### US-006：Prompt 批量测试

- [ ] 支持 CSV/JSON 格式上传测试用例
- [ ] 表格展示：序号 | 变量摘要 | 模型 | 质量分 | 状态
- [ ] 点击行展开完整 AI 输出（卡片形式）
- [ ] 支持多选行进行 A/B 对比
- [ ] 测试进度在任务列表中可见

### US-007：多轮对话流程测试

- [ ] 配置对话轮次数（默认 5 轮）
- [ ] 每轮输出作为下一轮输入自动拼接
- [ ] 可视化展示每轮的角色和内容
- [ ] 检测上下文累积导致的输出异常

### US-008：A/B 测试 + 统计显著性

- [ ] 支持同时对比 2 个 prompt 版本
- [ ] 序贯检验（SPRT）自动决定停止时机
- [ ] 最小 15 次、最大 50 次样本保护
- [ ] 显著性水平 α=0.05，功效 1-β=0.80
- [ ] 清晰展示：置信区间、p-value、胜出版本

### US-009：自动评测集生成

- [ ] AI 根据 prompt 自动生成 10 条基础评测 case
- [ ] 用户可对每条 case 编辑、删除、新增
- [ ] 支持设定评测集名称和描述
- [ ] 评测集可复用、导入导出

### US-010：Prompt 质量评分卡

- [ ] 四维度评分：Clarity(30%)、Completeness(30%)、Example(25%)、Role(15%)
- [ ] 权重可配置，不同 prompt 类型可保存不同权重模板
- [ ] 每维度给出分数 + 简短改进建议
- [ ] 雷达图可视化展示

### US-011：Prompt 回归检测

- [ ] 保存时触发：对比当前版本 vs 最新历史版本，运行前 5 条评测 case，30s 内返回
- [ ] 低于基线 15% 以上时告警（阻断保存可选）
- [ ] 模型升级时自动触发完整回归检测
- [ ] 告警结果记录在版本历史中

### US-012：结构化日志 + AI 调用链路追踪

- [ ] 每条日志包含 trace_id 贯穿全链路
- [ ] ai_call_log 表记录：token 消耗、延迟、model、cost
- [ ] 请求日志包含：method、path、status、latency、client_ip
- [ ] 高写入字段定期归档策略（7天后压缩）

### US-013：成本分析中心

- [ ] 月度 token 消耗趋势图（按 prompt 分组）
- [ ] 成本 Top 5 prompt 排名
- [ ] 按模型维度的费用分布饼图
- [ ] 导出月度成本报告（CSV）

### US-014：场景标签

- [ ] 支持一个 prompt 绑定多个场景标签
- [ ] 标签支持模糊搜索
- [ ] 标签管理页面：增删改、合并标签
- [ ] 标签使用统计（每个标签关联多少 prompt）

### US-015：Chain-of-Thought 引导器

- [ ] 提供 3-5 个 CoT 最佳实践模板
- [ ] 自动检测 prompt 是否缺少推理引导
- [ ] 一键将模板注入到当前 prompt

## 七、风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 统计显著性计算复杂度高 | 可能超出 MVP 时间 | 优先实现固定样本量 A/B，序贯检验作为可选特性 |
| AI 评测集生成质量不稳定 | 测试结果不可信 | 强制人工审核环节，不可跳过 |
| SQLite 在高并发下成为瓶颈 | 任务队列性能下降 | 预留架构：Worker 接口不变，后续可替换为 PostgreSQL |
| 变量实时预览性能 | 大 prompt 渲染卡顿 | 300ms debounce，大于 5000 字符时降级为手动刷新 |
